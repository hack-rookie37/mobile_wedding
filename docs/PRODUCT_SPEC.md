# PRODUCT_SPEC — 모바일 청첩장 빌더 (코드명: marriage)

- 문서 버전: 1.0 (2026-07-16)
- 상태: MVP 기준 확정. 가정(A-xx)은 §13 등록부에 기록하며, 변경 시 이 문서와 [DECISIONS.md](./DECISIONS.md)를 함께 갱신한다.

## 1. 제품 개요

한국 사용자를 위한 모바일 청첩장 제작 웹앱.

- **제작자**는 데스크톱 웹 편집기에서 청첩장을 만들고 발행한다.
- **하객**은 카카오톡/문자로 받은 링크를 열어 모바일 세로 스크롤 청첩장을 본다.
- AI가 한 번 만들어주고 끝나는 서비스가 아니다. 사용자가 버튼·입력창·드래그·슬라이더로 직접, 그리고 AI로도 계속 고칠 수 있는 **편집기**가 제품의 중심이다.

## 2. 사용자와 핵심 시나리오

### 제작자 (예비 신랑·신부)
1. 가입 → 새 청첩장 프로젝트 생성 → 기본 정보(이름·일시·장소) 입력.
2. 섹션을 추가/정렬/숨김하며 내용을 채우고, 중앙 미리보기에서 실제 모바일 화면을 확인.
3. 발행하기 → 공개 URL 획득 → 카카오톡/문자로 전달.
4. 발행 후에도 수정 → 재발행. RSVP 응답을 결과 페이지에서 확인.

### 하객
1. 링크 클릭 → 3G/LTE 환경에서도 빠르게 열리는 세로 스크롤 청첩장 열람.
2. 지도 앱 열기, 전화/문자, 계좌번호 복사, 갤러리 감상.
3. RSVP 섹션에서 참석 여부 제출.

## 3. 제품 원칙 (요구사항 10개의 규범화)

| # | 원칙 | 구현 상의 의미 |
|---|------|----------------|
| 1 | 지속 편집 가능한 도구 | 일회성 AI 생성물이 아닌 편집기 중심 제품 |
| 2 | 직접 조작 우선 | 모든 편집은 GUI 컨트롤로 가능해야 함. AI는 보조 수단 |
| 3 | section/block 기반 | 자유 좌표 배치 없음. 섹션 단위 구조 편집 |
| 4 | 실물 미리보기 | 편집기 중앙 = 실제 모바일 청첩장 (360/390/430px) |
| 5 | 단일 renderer | 미리보기와 공개 페이지가 같은 렌더러 모듈 사용 (ADR-004) |
| 6 | versioned JSON document | 청첩장 1개 = `schemaVersion`을 가진 JSON 문서 (ADR-002) |
| 7 | typed action 단일 파이프라인 | 직접 편집과 AI 편집이 같은 action을 dispatch (ADR-003) |
| 8 | undo/redo | 문서를 바꾸는 모든 action은 되돌릴 수 있음 |
| 9 | 데이터 분리 | RSVP 응답·공개 스냅샷은 draft 문서와 별도 저장 (ADR-006) |
| 10 | AI 프라이버시 경계 | 사진 원본·계좌·연락처·RSVP는 기본적으로 AI에 미전송 (§9, ADR-011) |

## 4. 정보 구조: 문서와 섹션

청첩장 1개 = **1개의 versioned JSON document**. 상세 스키마는 [ARCHITECTURE.md §3](./ARCHITECTURE.md) 참조.

```
document
├─ schemaVersion: 1
├─ wedding   ← 전역 공통 데이터 (신랑·신부·혼주, 예식 일시, 예식장)
├─ theme     ← 전역 디자인 (프리셋, 폰트 페어, 색)
└─ sections[] ← 순서 있는 섹션 목록 {id, type, visible, content, layout, style}
```

**전역 데이터 정규화 (발견된 공백 #7):** 신랑·신부 이름, 예식 일시, 장소는 Hero·Calendar·Venue 등 여러 섹션에 등장한다. 섹션마다 중복 저장하면 수정 누락이 생기므로 문서 전역 `wedding`에 단일 저장하고 섹션은 참조만 한다. 섹션별 `content`에는 해당 섹션 고유 정보만 둔다.

**섹션 인스턴스 규칙 (발견된 모순 #2):** 좌측 패널에 "복제"가 있으므로 같은 타입의 섹션이 여러 개 존재할 수 있어야 한다. 따라서 **섹션 타입은 다중 인스턴스 허용**하되, `hero`와 `rsvp`는 1개만 허용(복제·추가 버튼 비활성)한다. (A-05, A-06)

## 5. 섹션별 사양 (MVP 12종)

공통 필드 — 모든 섹션이 가진다:

| 필드 | 값 | 비고 |
|------|----|------|
| `visible` | boolean | 숨김 시 미리보기·공개 페이지에서 제외 |
| `layout.variant` | 타입별 enum | 아래 각 섹션 참조 |
| `style.paddingY` | `sm(48px)` / `md(72px)` / `lg(104px)` | 섹션 상하 여백 |
| `style.background` | hex (optional) | 미지정 시 테마 paper 색 |
| `style.animation` | `none` / `fade` / `rise` | 스크롤 진입 애니메이션 |

### 5.1 Hero (`hero`) — 항상 최상단 고정 (A-05)
- content: 대표 사진(assetId, optional), 태그라인(예: "Wedding Invitation"), 이름·일시·장소 표시 토글 3개 (값은 `wedding` 참조)
- variant: `photoFull`(전면 사진) / `photoArch`(아치 프레임) / `textOnly`
- 제약: 삭제·이동·복제·숨김 불가

### 5.2 Greeting (`greeting`)
- content: 제목, 본문(멀티라인, 개행 보존), 혼주 표기 토글(양가 부모 성함 + 서열 라벨, 故 표기 지원 — 값은 `wedding` 참조), 정렬(center/left)

### 5.3 Couple Profile (`coupleProfile`)
- content: 신랑/신부 각 {사진(assetId), 소개 문구}, 이름은 `wedding` 참조
- variant: `sideBySide` / `stacked`

### 5.4 Calendar (`calendar`)
- content: 달력 그리드 표시 토글, D-day 카운트다운 토글. 날짜는 `wedding.datetime` 참조
- 양력만 지원, 음력 표기는 post-MVP (A-15)

### 5.5 Gallery (`gallery`)
- content: photos[] {assetId, alt?} — 드래그로 순서 변경, 최대 30장 (A-14)
- variant: `grid2` / `grid3` / `carousel`
- 옵션: 라이트박스(탭하여 크게 보기, 스와이프) on/off

### 5.6 Video (`video`)
- content: YouTube URL, 제목(optional) — **외부 임베드 전용, 파일 업로드 없음** (A-07, 모순 #3 해소)
- 자동재생 금지(모바일 데이터 보호), 탭하여 재생

### 5.7 Venue (`venue`)
- content: 지도 표시 토글, 지도 확대 레벨, 길찾기 버튼 토글(카카오내비/네이버지도/티맵 딥링크). 예식장 이름·홀·주소·전화는 `wedding.venue` 참조
- 지도: Kakao Maps JS SDK (A-08, ADR-013)

### 5.8 Transportation (`transportation`)
- content: items[] {icon: `subway`/`bus`/`car`/`parking`/`shuttle`/`etc`, 제목, 설명(멀티라인)} — 반복 그룹 편집기

### 5.9 Contacts (`contacts`)
- content: 신랑측/신부측 그룹 각각 persons[] {역할 라벨(신랑/아버지/어머니/직접입력), 이름, **전화번호(sensitive)**}
- 게스트 뷰: 전화/문자 버튼. variant: `inline` / `accordion`(접기)

### 5.10 Gift Account (`giftAccount`)
- content: 신랑측/신부측 그룹 각각 {그룹 제목, accounts[] {은행, **계좌번호(sensitive)**, 예금주, **카카오페이 송금 링크(sensitive, optional)**}}
- 게스트 뷰: 아코디언 접힘 기본, "복사" 버튼(은행+계좌번호 클립보드 복사 → 토스트)

### 5.11 RSVP (`rsvp`)
- content: 안내 문구, 마감일(optional), 수집 필드 토글 {측 구분, 동행 인원, 식사 여부, 연락처, 메시지}
- 성명·참석 여부는 항상 수집. **개인정보 수집·이용 동의 체크 필수** (A-16, §9)
- 게스트 제출 데이터는 문서가 아닌 별도 테이블에 저장 (원칙 9)

### 5.12 Closing (`closing`)
- content: 마무리 문구, 사진(assetId, optional), 공유 버튼 토글(링크 복사 + Web Share API)

## 6. 편집기 사양 (데스크톱, 최소 뷰포트 1280px — A-03)

```
┌────────────────────────────────────────────────────────────┐
│ 상단바: 프로젝트명(인라인 수정) · 저장상태 · ↶ ↷ · 미리보기 · [발행하기] │
├──────────┬──────────────────────────────┬──────────────────┤
│ 좌측 264px │        중앙 (가변)            │   우측 320px      │
│           │  ┌─ 360|390|430 ─ 편집|인터랙션 ┐ │                  │
│ [기본 정보] │  │                          │ │  선택 컨텍스트     │
│ [테마]     │  │   실제 모바일 청첩장       │ │  ┌─────┬─────┐  │
│ ────────  │  │   미리보기 (실스크롤)      │ │  │콘텐츠│디자인 │  │
│ 섹션 목록   │  │                          │ │  └─────┴─────┘  │
│  ⠿ Hero   │  └──────────────────────────┘ │  선택 섹션의 폼    │
│  ⠿ 인사말  │                               │                  │
│  …        │                               │                  │
│ [+ 섹션 추가]│                              │                  │
└──────────┴──────────────────────────────┴──────────────────┘
```

### 상단바
- **프로젝트명**: 클릭하여 인라인 수정 (undo 비대상 — 문서 외부 데이터, ADR-003)
- **저장 상태**: `저장 중…` / `모든 변경사항 저장됨` / `저장 실패 — 재시도` 3상태. 자동저장은 마지막 입력 후 1.5초 디바운스 (A-12)
- **undo/redo**: 버튼 + `⌘Z` / `⇧⌘Z` (Windows `Ctrl`). 히스토리는 세션 인메모리 최대 100스텝 (A-11)
- **미리보기**: 새 탭에서 소유자 전용 draft 뷰(`/preview/[projectId]`) — 발행 전 확인용 (A-19)
- **발행하기**: 모달 → 검증 결과 표시 → 발행 → URL + 복사 버튼 (§7)

### 좌측 패널
- 최상단 고정 항목 2개: **기본 정보**(wedding 데이터 폼), **테마**(전역 디자인) — 섹션이 아닌 전역 데이터의 편집 진입점 (공백 #8 해소)
- 섹션 목록: 드래그 핸들(⠿)로 순서 변경, 눈 아이콘으로 숨김 토글, ⋯ 메뉴에서 복제/삭제
- `+ 섹션 추가`: 12종 카탈로그 모달. hero·rsvp는 이미 존재하면 비활성 (A-06)
- 삭제는 확인 없이 실행하되 undo로 복원 가능 (스냅바에 "실행 취소" 노출)

### 중앙 미리보기
- 디바이스 프레임 안에 **공개 페이지와 동일한 renderer**가 그대로 마운트 (원칙 5)
- 폭 전환: 360 / 390 / 430px
- **모드 토글 (발견된 모순 #1 해소)**: "실제 인터랙션 확인"과 "클릭으로 섹션 선택"이 충돌하므로 두 모드를 둔다.
  - `편집 모드`(기본): hover 시 섹션 아웃라인, 클릭 시 해당 섹션 선택 → 우측 패널 열림. 갤러리 스와이프 등 내부 인터랙션은 차단
  - `인터랙션 모드`: 게스트와 동일하게 동작(스와이프, 아코디언, 복사 버튼 등). 선택 불가
- 스크롤은 두 모드 모두 실제 스크롤
- 선택 동기화: 좌측 목록 클릭 ↔ 미리보기 해당 섹션으로 스크롤·하이라이트

### 우측 패널
요구된 6개 영역(내용/레이아웃/스타일/사진/여백/애니메이션)을 2개 탭으로 그룹핑:
- **콘텐츠 탭**: 내용 필드 + 사진 설정 (섹션 타입별 폼, §5)
- **디자인 탭**: layout.variant(세그먼트 컨트롤) + 여백(paddingY) + 배경색 + 애니메이션 preset
- 기본 정보/테마 선택 시 해당 전역 폼 표시. 아무것도 선택 안 하면 "섹션을 선택하세요" 빈 상태

## 7. 발행과 공개 페이지

### 발행 모델 (모순 #5 해소, ADR-012)
- **draft**(편집 중 문서)와 **publication**(발행 스냅샷)을 분리 저장한다.
- 발행 시 draft를 검증 후 스냅샷으로 복사. 이후 draft를 수정해도 공개 페이지는 **재발행 전까지 불변**.
- 프로젝트당 라이브 스냅샷 1개. 재발행은 같은 URL을 덮어쓴다. 발행 이력 보관은 post-MVP.
- **발행 중단** 가능: 공개 페이지가 "청첩장이 비공개 처리되었습니다" 안내로 전환.

### 발행 검증 (fail fast)
| 규칙 | 수준 |
|------|------|
| 신랑·신부 이름, 예식 일시, 예식장 이름·주소 존재 | 오류(발행 차단) |
| 보이는 섹션 1개 이상 | 오류 |
| RSVP 마감일이 예식일 이후 | 오류 |
| 예식 일시가 과거 | 경고(발행 허용) |
| 사진 없는 photoFull Hero | 경고 |

### 공개 페이지
- URL: `/i/[slug]` — slug는 추측 곤란한 랜덤 10자 (A-09). 커스텀 slug는 post-MVP
- 미로그인 접근, 검색엔진 비색인(`noindex` — 사적 문서)
- OG 메타태그(제목 "신랑 ♥ 신부 결혼합니다", 대표 사진, 일시) → 카카오톡/문자 링크 미리보기 (A-18). 카카오 SDK 공유 버튼은 VS7
- 존재하지 않는 slug → 404, 발행 중단 → 안내 페이지

## 8. RSVP

### 게스트 폼 (A-16)
- 필수: 성명, 참석 여부(참석/불참), **개인정보 수집·이용 동의**
- 토글 필드(제작자 설정): 측 구분(신랑측/신부측), 동행 인원(본인 제외 숫자), 식사 여부, 연락처, 메시지
- 제출 후: 감사 화면으로 전환, 같은 브라우저 재제출 방지(localStorage 소프트 가드)
- 마감일 경과 또는 발행 중단 시 폼 비활성

### 제작자 결과 뷰 (공백 #9 해소, A-22)
- 경로: `/projects/[projectId]/rsvp` (편집기 밖 별도 페이지)
- 집계 카드(참석/불참/동행 합계, 측별·식사별) + 응답 목록 테이블 + CSV 다운로드
- 응답 개별 삭제 가능(게스트 요청 대응)

### 스팸 방어 (A-17)
- MVP: 허니팟 필드 + 프로젝트별 일일 제출 상한. 캡차는 post-MVP

## 9. 개인정보·AI 데이터 정책

**두 가지 축을 구분한다 — "게스트에게 공개"와 "AI 모델에 전송"은 다른 문제다.**
연락처·계좌번호는 청첩장의 본질상 게스트에게는 보이지만, AI 모델에는 기본적으로 보내지 않는다.

| 데이터 | 게스트 공개 | AI 전송 (기본) | 저장 위치 |
|--------|------------|---------------|----------|
| 인사말·일정·장소 등 일반 콘텐츠 | ✓ | ✓ | 문서(draft/snapshot) |
| 전화번호 (contacts) | ✓ (버튼) | ✗ redact | 문서 — `sensitive` 마킹 |
| 계좌번호·카카오페이 링크 | ✓ (복사) | ✗ redact | 문서 — `sensitive` 마킹 |
| 사진 원본/URL | ✓ (변환본) | ✗ (assetId·치수·방향 메타만 — 구현: ARCHITECTURE §10) | Storage + assets 테이블 |
| RSVP 응답 | ✗ (소유자만) | ✗ (절대 미전송) | 별도 테이블 |

- **redaction 규칙 (모순 #4 해소)**: AI에는 `sensitive` 필드를 구조 보존 placeholder(`<redacted>`)로 치환한 문서 사영(projection)을 전달한다. AI는 해당 섹션의 구조 편집(순서·표시 여부)은 할 수 있으나 민감 값을 읽을 수 없다. 단, **사용자가 채팅에 직접 입력한 값**(예: "계좌번호를 110-xxx로 바꿔줘")은 사용자 제공 데이터이므로 action으로 반영 가능하다.
- 업로드 사진은 클라이언트에서 재인코딩하여 **EXIF(GPS 포함) 제거** 후 저장. 원본 raw 파일은 보관하지 않는다 (A-13)
- RSVP 게스트 개인정보(공백 #13): 폼에 수집 항목·목적·보관 기간 고지 + 동의 체크. 소유자에게 예식 후 삭제 안내. 자동 파기는 post-MVP
- 상세 기술 경계는 [ARCHITECTURE.md §10](./ARCHITECTURE.md), [DECISIONS.md ADR-011](./DECISIONS.md)

## 10. 성능·접근성 목표 (공개 페이지)

| 항목 | 목표 |
|------|------|
| LCP | ≤ 2.5s (중급 안드로이드, 4G 스로틀) |
| CLS | ≤ 0.1 (이미지 크기 예약 필수) |
| INP | ≤ 200ms |
| 초기 JS | ≤ 130KB gzip |
| 이미지 | 반응형 변환(AVIF/WebP), 뷰포트 밖 lazy, Hero는 priority |
| 폰트 | 한글 서브셋 woff2 self-host ([DESIGN_SYSTEM.md §4](./DESIGN_SYSTEM.md)) |

접근성: 본문 대비 4.5:1 이상, 터치 타깃 44×44px, 모든 사진 alt, `prefers-reduced-motion` 시 애니메이션 제거, 시맨틱 랜드마크/헤딩 구조.

## 11. MVP에서 만들지 않는 것

요구사항 명시 6개 + 이 문서에서 후순위로 정리한 것:

- 자유 좌표 배치, 실시간 다중 사용자 편집, 네이티브 앱, 영상 트랜스코딩, 결제, 템플릿 마켓플레이스 (요구사항)
- 커스텀 slug/도메인, 발행 이력·롤백, 음력 표기, 방문 통계, 캡차, RSVP 자동 파기, 게스트 방명록, 다국어, 모바일 편집기, 외부 공유용 draft 토큰 링크, 카카오 알림톡
